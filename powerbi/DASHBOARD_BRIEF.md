# Power BI Dashboard — Build Brief

**Project:** Customer Lifetime Value & Churn Prediction
**Deliverable:** A three-page Power BI report, portfolio quality
**Audience:** Recruiters and hiring managers assessing an intermediate data analyst

---

## 1. What you are building and why

A telecommunications provider has 7,043 customers. A model predicts which are likely to leave. A separate piece of work values each customer. This report presents the finding that comes from putting those two together.

**The finding, in one sentence:** acting on the churn model alone loses money; acting on it selectively makes money; and the entire difference comes from deciding who to leave alone rather than from any improvement to the model.

The report must make that argument visible in about ten seconds on page one, then let a reader interrogate it on pages two and three. It is a portfolio piece, so it is judged on whether the reasoning is legible and the presentation disciplined — not on how many visuals it contains.

---

## 2. Business context you need

All figures below are already computed and sitting in the data. You are not calculating them, you are displaying them. Full narrative is in `md/01`–`md/04_*_writeup.md` if more depth is needed.

**Customer value.** Every customer carries a value (`clv`). The base is worth **R2.73 million**. Customers are ranked into ten equal value groups (`value_decile`, 1 = most valuable).

**The model.** A logistic regression estimates each customer's probability of leaving (`churn_probability`). It was chosen over gradient boosting because its reasoning can be inspected, at no cost in performance. Customer value was deliberately kept out of the model, so risk and worth stayed separate questions.

**The decision rule.** For each customer: `probability of leaving × 30% chance of retention × customer value − R60 contact cost`. Positive means contact is worth it. The R60 is R20 admin plus a R40 offer. The 30% is the least certain input in the project.

**The result.** 2,218 customers qualify, returning **+R39,662** — a 30% return on spend. The other 4,825 would cost R289,500 to contact and return R105,819, destroying **R183,681**. Contacting everyone loses **R144,018**. The swing between targeting and not targeting is roughly **R184,000**, with an identical model in both cases.

**Three findings worth surfacing.** Value groups 8–10 (2,112 customers) qualify *nobody* — they are not worth keeping at any price. Groups 4 and 5 (1,408 customers) produce 55% of the entire return from 20% of the base. Only 83 of the 705 most valuable customers qualify, because they are not leaving — so aiming retention at your best customers, the instinctive move, loses money here.

**An honest caveat to display, not bury.** The programme is fragile. Return is R39,662 on a R2.73m base — small, and it should be described as small. Below about a 25% retention success rate the programme stops being worth running at all.

## 4. Measures

Keep all business logic in SQL. Measures must be aggregation only — `SUM`, `COUNTROWS`, `DIVIDE`, and `CALCULATE` filtering on columns SQL already computed. Do not recreate the R60 cost, the 0.5 threshold, or the decision rule in DAX. They are columns already.

**Scale:** `Customers` = COUNTROWS · `Book Value` = SUM(`clv`) · `Value at Risk` = SUM(`value_at_risk`)

**Decision:** `Targeted Customers` and `Customers Left Alone` = COUNTROWS filtered on `decision` · `Share Targeted` = DIVIDE · `Net if Targeted` = SUM(`net_value`) where `decision = "Intervene"` · `Net if Contact Everyone` = SUM(`net_value`) unfiltered · `Value Destroyed by Untargeted Contact` = SUM(`net_value`) where `decision = "Do not intervene"` · `Decision Swing` = `[Net if Targeted] − [Net if Contact Everyone]` · `Value Recovered` = SUM(`expected_value_saved`) where Intervene · `Contact Cost` = SUM(`intervention_cost`) where Intervene · `Return on Spend` = DIVIDE(Net if Targeted, Contact Cost)

**Model — every one filtered to `data_split = "test"`:** `Test Customers` · `Actual Leavers` · `Caught` · `Missed` · `False Alarms` · `Correctly Left Alone` (all COUNTROWS filtered on `outcome_class`) · `Recall` = Caught ÷ (Caught + Missed) · `Precision` = Caught ÷ (Caught + False Alarms) · `Accuracy` = (Caught + Correctly Left Alone) ÷ Test Customers · `Baseline Accuracy` = (Test Customers − Actual Leavers) ÷ Test Customers

**Sensitivity:** simple SUMs over the eight rows. `Sens Return on Spend` divides `return_on_spend_pct` by 100.

Currency format `"\R#,0"`. Percentages `"0.0%"`.

## 5. The three pages

Canvas 1440×810. Consistent 32px outer margin, 12–16px gutters.

### Page 1 — Executive Summary

Make the argument. A reader should leave knowing that targeting beats blanket contact by ~R184,000 and that the model is identical in both cases.

- **Title block**, top left: "Retention economics" with a one-line subtitle naming the 7,043 customers.
- **KPI row**, five cards: Book Value · Targeted Customers · Net if Targeted (positive colour) · Net if Contact Everyone (negative colour) · Decision Swing. The third and fourth sitting side by side, one positive and one negative, is the whole story in two numbers.
- **Return breakdown**, left: Value Recovered → Contact Cost → Net if Targeted. A waterfall if available, otherwise a column chart.
- **Sensitivity curve**, right: line chart, `success_rate` on X, `Sens Net Targeted` and `Sens Net Everyone` as two series. Mark the 30% assumption. This visual is what stops the headline number reading as a claim.
- **Finding**, full width across the bottom: a prose statement with the two money figures colour-coded inline, plus a smaller line stating both assumptions and the ~25% break-even.

### Page 2 — Segment Drill-Down

Where the return sits and who to leave alone.

- **Slicers** across the top: `contract`, `tenure_band`, `spend_band`. These drive everything on the page.
- **Hero chart**: clustered column by `value_decile`, two series — `Net if Targeted` against `Net if Contact Everyone`. Groups 8–10 sit at zero on one series and deeply negative on the other. That contrast is the page's main point; make it legible.
- **Matrix** by `value_decile`: Customers, Targeted Customers, Net if Targeted. Turn subtotals off.
- **Churn reasons**: horizontal bar of `churn_reason` by count. **Label it as recorded after departure and excluded from the model.**
- **Map**: bubbles on `latitude`/`longitude`, sized by Targeted Customers. Supporting visual, not a hero. If it reads as decoration once rendered, cut it — it is the most expendable thing on the page.
- **Annotation panel**: the three findings from §2, briefly.

### Page 3 — Model Performance

The model's honest record.

- **Title** plus a prominent note that every figure comes from the 1,761 held-back customers only.
- **KPI row**: Recall · Precision · Accuracy · Baseline Accuracy · Test Customers.
- **Confusion matrix** as four cards: Caught 368 · Missed 99 · False alarm 361 · Correctly left alone 933. Positive colour for the two correct cells, negative for the two errors, with text labels so colour is never doing the work alone.
- **Outcome chart**: counts by `outcome_class`.
- **Explanatory panel**, which must cover three things. ROC AUC is 0.838, inside the expected range — near 1.0 would suggest leakage. Accuracy of 73.9% sits barely above the 73.5% available from predicting nobody ever leaves, so on accuracy alone the model looks worthless while in fact finding nearly four in five leavers; this is why precision and recall are reported instead. And roughly one flag in two is a customer who was not leaving, which is not a fault to fix but the limit of predicting departure from a profile — it is priced into the economics rather than argued around.

## 6. Rules that must not be broken

**Page 3 uses held-back customers only.** Every model metric filters `data_split = "test"`. Measuring across all 7,043 includes customers the model trained on and overstates it. The write-ups state this limitation explicitly; contradicting it on the dashboard would be worse than omitting the page.

**Use the selected model's figures: 368 caught, 99 missed, 361 false alarms, recall 78.8%, precision 50.5%, AUC 0.838.** An earlier model version scored 370 / 97 / 362 and recall 79.2% at AUC 0.840. Those figures appear in Stage Three's write-up and in older notes, but they belong to a version that was rejected because its coefficients were unreliable. The stored probabilities come from the selected model. Mixing the two is the single easiest mistake to make here — worth a footnote on the page explaining the discrepancy rather than leaving a reader to find it.

**`churn_reason` never feeds anything predictive.** Drill-down and description only.

**No business logic in DAX.** If a number needs a rule to compute, that rule belongs in SQL and the column already exists.

**Do not relate the two tables.**

## 7. Design

Judged on restraint. Deliberate and consistent beats decorated.

**Palette** — a dark surface with a diverging pair carrying the central idea, value created against value destroyed. These specific values are validated for colourblind separation (protan ΔE 19.2) and for contrast against the dark surface:

| Role | Hex |
|---|---|
| Page background | `#0d0d0d` |
| Card / visual surface | `#1a1a19` |
| Primary text | `#ffffff` |
| Secondary text | `#c3c2b7` |
| Muted — axes, labels | `#898781` |
| Gridlines, borders | `#2c2c2a` |
| **Value created** | `#3987e5` |
| **Value destroyed** | `#e66767` |

Blue/red rather than green/red: it survives colourblindness, where green/red does not.

**Rules.** One accent pair, used consistently — blue always means money made, red always means money lost, on every page. Never a dual-axis chart. No pie charts, gauges, or 3D. Gridlines recessive. Labels selectively, never a number on every point. Text in text colours, never in the series colour. Where colour marks a category, a text label carries it too. Segoe UI throughout. Consistent card treatment, generous whitespace, aligned edges.

**Titles do work.** "Net return by value group, targeted against contacting everyone" beats "Net Value by Decile". A subtitle should say what the reader is meant to notice.

## 9. A note on how to build this

Power BI's `.pbix` is a binary format that cannot be authored from outside the application. Two routes exist:

**Author the `.pbip` project as files.** Power BI Desktop opens this text-based format natively, and it is the only route available to a tool that can read and write files but cannot operate a GUI. A partial implementation exists at `powerbi/ChurnEconomics.pbip` with generator scripts in `powerbi/generate/` — `gen_model.py` builds the TMDL semantic model, `gen_report.py` builds the PBIR report. **It does not currently open.** The schema URLs are wrong; `find_schemas.py` extracts the correct ones from the installed Power BI binaries and is the reliable way to settle them rather than guessing. The semantic model, all 27 measures, and all three page layouts are already written and cross-checked.

**Or build it by hand in Power BI Desktop**, using this brief as the specification.

An agent that can only run commands and edit files cannot click, drag, or position visuals on the Power BI canvas, and cannot see what rendered. Whichever route is taken, a human needs to open the finished file and confirm it looks right — that verification cannot be delegated.
